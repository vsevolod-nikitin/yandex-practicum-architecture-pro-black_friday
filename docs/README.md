## Проектирование схем коллекций для шардирования данных (задание 7)

Онлайн-магазин «Мобильный мир» хранит заказы, товары и корзины в трёх коллекциях MongoDB. Ниже — схемы коллекций, анализ кандидатов на шард-ключ, выбранная стратегия шардирования и примеры команд.

---

### orders

Хранение заказов клиентов.

| Поле | Тип | Описание |
|------|-----|----------|
| `_id` | ObjectId | Уникальный идентификатор заказа |
| `client_id` | ObjectId | Идентификатор клиента |
| `created_date` | Date | Дата и время оформления заказа |
| `items` | Object[] | Список товаров. Массив документов `{ product_id: ObjectId, product_name: String, quantity: Number, price: Decimal128 }` |
| `status` | String | Статус заказа: `"pending"` \| `"paid"` \| `"shipped"` \| `"delivered"` \| `"cancelled"` |
| `total_amount` | Decimal128 | Общая сумма заказа |
| `geozone` | String | Геозона заказа (например, `"moscow"`, `"ekb"`) |

**Основные операции:** быстрое создание заказа, поиск истории заказов пользователя, отображение статуса заказа.

#### Кандидаты на шард-ключ

| Кандидат | Плюсы | Минусы |
|----------|-------|--------|
| `_id` | Равномерное распределение | Типовые запросы идут по `client_id`, а не по `_id` |
| `client_id` | Запросы истории и статуса заказа по клиенту | Возможен hot shard для очень активных клиентов |
| `geozone` | Логично для региональной аналитики | Низкая кардинальность, неравномерное распределение (Москва >> остальные) |
| `created_date` | Удобно для временных срезов | Низкая кардинальность, все новые заказы попадают на один шард |

#### Выбранная стратегия

**Шард-ключ:** `{ client_id: "hashed" }`

**Обоснование:** основные операции фильтруют данные по `client_id` — создание заказа, получение истории заказов и проверка статуса выполняются как запросы на один шард. Хэширование по `client_id` обеспечивает равномерное распределение при большом числе клиентов. Поля `geozone` и `created_date` отвергнуты как шард-ключи из-за низкой кардинальности и риска неравномерной нагрузки.

**Вторичные индексы:**

```javascript
db.orders.createIndex({ client_id: 1, created_date: -1 })
db.orders.createIndex({ client_id: 1, status: 1 })
```

---

### products

Сведения о товарах.

| Поле | Тип | Описание |
|------|-----|----------|
| `_id` | ObjectId | Уникальный идентификатор товара |
| `name` | String | Наименование |
| `category` | String | Категория товара |
| `price` | Decimal128 | Цена |
| `remains` | Object | Остаток товара в каждой геозоне. Объект вида `{ moscow: 50, ekb: 30 }` |
| `attributes` | Object | Дополнительные атрибуты: `{ color: String, size: String, ... }` |

**Основные операции:** частые обновления остатков при покупках, поиск по категориям и фильтрация по цене, описание товара на странице продукта.

#### Кандидаты на шард-ключ

| Кандидат | Плюсы | Минусы |
|----------|-------|--------|
| `_id` | Запросы страницы товара и обновления остатков; равномерное распределение | Поиск по категории/цене |
| `category` | Запросы каталога по категории | Hot shard для популярных категорий («Электроника») |

#### Выбранная стратегия

**Шард-ключ:** `{ _id: "hashed" }`

**Обоснование:** самые частые write-операции — атомарное списание остатков при покупке и обновление карточки товара — выполняются по `product_id` (`_id`), что даёт запрос на один шард. Хэширование обеспечивает равномерное распределение каталога. Поиск по категории и диапазону цен обслуживается вторичным индексом. Шардирование по `category` отвергнуто из-за риска hot shard на популярных категориях.

**Вторичные индексы:**

```javascript
db.products.createIndex({ category: 1, price: 1 })
db.products.createIndex({ name: "text" })
```

### carts

Данные о текущих корзинах (гостевых и пользовательских).

| Поле | Тип | Описание |
|------|-----|----------|
| `_id` | ObjectId | Уникальный идентификатор корзины |
| `owner_id` | String | Единый идентификатор владельца корзины: `user_id` (для авторизованных) или `session_id` (для гостей). Всегда заполнен |
| `user_id` | ObjectId \| null | Идентификатор пользователя (null для гостей) |
| `session_id` | String \| null | Идентификатор сессии (null для авторизованных пользователей) |
| `items` | Object[] | Список товаров. Массив документов `{ product_id: ObjectId, quantity: Number }` |
| `status` | String | Статус корзины: `"active"` \| `"ordered"` \| `"abandoned"` |
| `created_at` | Date | Дата и время создания |
| `updated_at` | Date | Дата и время последнего обновления |
| `expires_at` | Date | Время автоматического удаления (TTL) |

**Основные операции:** создание корзины, получение активной корзины по `session_id` или `user_id`, добавление/удаление товаров, слияние гостевой корзины в пользовательскую, отметка корзины как заказанной.

> Поле `owner_id` вычисляется при создании/обновлении корзины (`user_id ?? session_id`) и используется как шард-ключ.

#### Кандидаты на шард-ключ

| Кандидат | Плюсы | Минусы |
|----------|-------|--------|
| `_id` | Равномерное распределение | Корзину ищут по `user_id`/`session_id`, а не по `_id` |
| `user_id` | Для авторизованных | Не работает для гостевых корзин |
| `session_id` | Для гостей | Не работает для пользовательских корзин |
| `owner_id` | Для всех типов корзин; равномерное распределение | Требует дополнительного поля в схеме |

#### Выбранная стратегия

**Шард-ключ:** `{ owner_id: "hashed" }`

**Обоснование:** все ключевые операции фильтруют корзину по одному идентификатору владельца — получение активной корзины, добавление товаров, слияние гостевой корзины при логине и отметка как заказанной. Поле `owner_id` всегда заполнено, поэтому хэширование даёт равномерное распределение без скопления документов с `null`.

**Вторичные индексы:**

```javascript
db.carts.createIndex({ owner_id: 1, status: 1 })
```

## Выявление и устранение «горячих» шардов (задание 8)

Онлайн-магазин «Мобильный мир» столкнулся с перегрузкой одного из шардов MongoDB: около 70% запросов приходилось на товары категории «Электроника». Ниже — метрики для раннего обнаружения дисбаланса, механизмы автоматического перераспределения данных и стратегия предотвращения подобных инцидентов.

---

### Контекст и причина инцидента

**Симптом:** один шард обрабатывает непропорционально большую долю read/write-операций, растут задержки и CPU.

**Вероятная причина:** шард-ключ основан на поле `category` (или другом ключе с низкой кардинальностью). Популярная категория «Электроника» сконцентрировала данные и трафик на одном шарде.

> В задании 7 шардирование по `category` было отвергнуто из-за риска hot shard. Целевая архитектура — `{ _id: "hashed" }` для коллекции `products`.

---

### Метрики мониторинга

#### 1. Распределение данных по шардам

```javascript
db.products.getShardDistribution()
```

#### 2. Нагрузка и задержки

```javascript
db.serverStatus().opcounters              // insert, query, update, delete
db.serverStatus().opLatencies.reads
db.serverStatus().opLatencies.writes
db.serverStatus().connections.current
```

#### 3. Состояние балансировщика и миграций

```javascript
sh.getBalancerState()
sh.isBalancerRunning()
```

#### 4. Системные метрики (на уровне ОС / Prometheus)

- CPU, RAM, disk I/O, network

#### Пороги и алерты

| Метрика | Порог | Действие |
|---------|-------|----------|
| Разница `opcounters.query` между шардами | > 30% от среднего | алерт, анализ hot category |
| p95 `opLatencies.reads` на шарде | > 50 ms | split/move чанков, масштабирование |
| Доля чанков коллекции на одном шарде | > 35% | проверить балансировщик, zone sharding |
| CPU шарда | > 80% в течение 5+ мин | изоляция hot category, кэширование |
| Доля запросов на одну категорию | > 50% read-трафика | кэш каталога, zone sharding |

---

### Механизмы автоматического перераспределения данных

#### 1. Встроенный балансировщик MongoDB

Основной механизм: балансировщик периодически переносит чанки с перегруженных шардов на менее загруженные.

```javascript
sh.setBalancerState(true)
sh.startBalancer()

// окно балансировки — вне пиковых часов
sh.setBalancerWindow("23:00", "06:00")

```

#### 2. Ручное и скриптовое управление чанками

```javascript
// разделить чанк по значению шард-ключа
sh.splitAt("mobile_world.products", { category: "electronics", _id: ObjectId("...") })

// разделить по результату find (для больших чанков)
sh.splitFind("mobile_world.products", { category: "electronics" })

// переместить чанк на другой шард
sh.moveChunk(
  "mobile_world.products",
  { category: "electronics" },
  "shard02"
)
```

---

### Стратегия предотвращения

#### Смена шард ключа
**Шард-ключ:** `{ _id: "hashed" }`

| | |
|---|---|
| **Плюсы** | Равномерное распределение каталога |
| **Минусы** | Медленный поиск по категории; компенсируется вторичным индексом и кэшем |


#### Оперативная мера без смены шард-ключа

- **Кэш каталога** (Redis) по ключу `category:electronics`.
- **Read preference** `secondaryPreferred` для аналитических запросов по категориям — разгружает primary.

#### Альтернатива: составной ключ `{ category: 1, geozone: 1 }`

Подходит для **регионального** дисбаланса (Москва vs малые города), но **не решает** hot shard внутри одного региона: в Москве «Электроника» по-прежнему сконцентрирует нагрузку.

## Настройка чтения с реплик и консистентность (задание 9)

Онлайн-магазин «Мобильный мир» использует replica set для коллекций `orders`, `products` и `carts` (схемы и операции — в задании 7). Ниже — распределение read-операций между primary и secondary, допустимая задержка репликации и обоснование с учётом консистентности, частоты обновлений и бизнес-логики. 

**Принцип:** если ошибка влияет на деньги, доступность товара или доверие пользователя сразу после действия — чтение только с primary; для каталога и истории допустима eventual consistency.

---

### Таблица операций чтения

| **Коллекция** | **Операция** | **Реплика чтения** | **Допустимая задержка** | **Обоснование** |
|:-------------:|:-------------|:-------------------|:------------------------|:----------------|
| orders | История заказов конкретного пользователя | secondary | до 30 сек | Список заказов меняется редко; устаревшая запись в истории не влияет на бизнес-операции. Read preference: `secondaryPreferred` |
| orders | Отображение статуса заказа (общий просмотр) | secondary | до 5 сек | Статусы `shipped` / `delivered` обновляются нечасто; небольшая задержка допустима для экрана «Мои заказы» |
| orders | Заказ сразу после оформления (страница подтверждения) | primary | — | Пользователь должен сразу увидеть созданный заказ и актуальный статус `pending` / `paid`; чтение с secondary может вернуть пустой результат или устаревший статус |
| products | Поиск товаров по категориям и фильтрация по диапазону цен | secondary | до 5 сек | Read-heavy операция; пользователь долго просматривает результаты, небольшая задержка каталога допустима. Финальная проверка — при добавлении в корзину и оформлении |
| products | Описание товара на странице продукта (название, цена, атрибуты) | secondary | до 5 сек | Описание и атрибуты обновляются редко; расхождение цены на несколько секунд некритично до момента покупки |
| products | Проверка остатков при добавлении в корзину и оформлении заказа | primary | — | Риск продажи недоступного товара; остатки в `geozones` часто обновляются при покупках |
| carts | Получение активной корзины по `owner_id` / `session_id` | primary | — | Корзина меняется при каждом действии пользователя; после оформления заказа статус должен быть `ordered`, а не `active` |
| carts | Чтение корзины при добавлении / удалении товара | primary | — | Нужны актуальные `items` и `quantity` для корректного обновления |
| carts | Чтение при слиянии гостевой и пользовательской корзины | primary | — | Обе корзины должны быть консистентны в момент слияния |
